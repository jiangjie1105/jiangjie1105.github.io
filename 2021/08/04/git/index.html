git的使用

git 目前最先进的分布式管理系统，用起来！！！

因为自己之前学习过git，本次就对一些git基本指令做一次复习。

时光机穿梭

git带来的体验是美妙的，所以我们可以体验到时光穿梭的感觉，对于工作区文件来说，我们甚至可以回到过去，改变某些事情，再穿梭回来，太神奇了！！

版本回退

如果多次添加文件或项目（git add）,如果想要知道自己想回退的是哪个版本的话，可以用git log查看日志。

首先，必须要知道当前的版本是哪个版本，在git中，用Head表示当前版本，上一个版本就是Head^,上上一个版本就是Head^^,以此类推，所以当我们想要回退版本时，可以使用Git reset命令:

此刻回到过去了，想回到未来怎么办？注意，只要控制台没关，之前的log日志还在的话，可以使用 git reset–hard (版本号) 就可以回到未来。

那如果命令台关了呢？

不要慌，这个时候可以用git reflog 命令， 它可以记录我们的每一次命令，顺便看到你想去版本的版本号，然后再使用 git reset 操作就可以了!

本节小结：

  • HEAD 指向的就是当前版本，因此可以在历史中穿梭，使用命令：git reset –hard commit_id
  • 穿梭前用git log 查看提交历史，以便确定回退到哪个版本。
  • 要重返未来，用git reflog查看命令历史，以便确定回到未来的哪个版本。

工作区和暂存区

很简单，git的版本库里面有很多东西，最重要的就是 stage （暂存区），和一个git默认为我们创建的master分支，以及指向master分支的一个指针叫HEAD。

管理修改

一句话概括，如果有两次修改，那么每一次修改后，都应该提交到暂存区（git add） 然后再进行提交（git commit）这样就不会发生第二次修改没被提交的情况啦。

撤销修改

当我们的修改没有达到满意的效果，想要撤销掉修改，怎么办？？
可以分情况来看：

  1. 按照上面的图来看：

    如果只是在工作区想要撤销，那么可以直接使用git checkout – file来撤销工作区的修改，这里需要主义的是 checkout后边一定是两个 –
    因为一个- 的意思是切换分支，这是两种完全不同的概念。
    总之: git checkout – file 就是把这个文件回到最近一次add 或者 commit 前 的状态。

2.如果已经提交到暂存区了（git add）那么可以使用 git reset HEAD (File)可以把暂存区的修改撤销掉，重新放回工作区。—-git reset命令既可以回退版本，也可以把暂存区的修改回退到工作区。回退到工作区后再进行一次 git checkout –file 后 就可以了。

  1. 如果已经commit 到了版本库，但是还没有提交到远程库，那么可以使用版本回退：git reset –hard HEAD^ （上一个版本）

删除文件

如果在工作区加了一个文件：比如 readme.txt 然后提交到了版本库中的暂存区。

然后在本地（也就是工作区删除了这个文件），这个时候不用担心，可以从版本库中恢复文件，还记得撤销操作吗？ 使用 git checkout – readme.txt 就可以回到删除之前的状态了。

git checkout其实是用版本库里的版本来替换工作区的版本，无论工作区是修改还是删除，都可以“一键还原”

分支管理（重要）

创建和删除分支

  • 创建分支： git branch dev(创建dev分支)
  • 切换分支： git checkout dev(切换到dev分支) 或者 git switch dev
    以上两个命令可以简写成 git checkout -b dev (即创建并切换到dev分支)
  • 查看分支： git branch
  • 合并分支： git merge dev(注意：意思是在master分支上合入dev分支，得先切换到master分支上)
  • 删除分支： git branch -d dev(删除dev分支)

解决冲突

如果两个分支都改动并提交了，merge时就可能发生冲突，解决冲突的办法就是把git合并失败的分支手动编辑为我们需要的内容，再提交。 用git log –graph命令可以看到分支合并图。

BUG分支

修改bug时使用，当接到一个代号101的修bug任务，本能的会想到新建一个issue 分支来修复bug，但是，当前正在dev上进行的工作还没有提交，并不是不想提交，而是工作进行了一半，还没法提交，比如预计还需一天才提交，但是bug要在两小时内完成，怎么办？？？？

这个时候，可以用到Stash功能，可以把现场的工作“储藏”起来，等恢复后继续工作：

git stash 后 再使用git status 查看工作区 就可以发现，工作区是干净的。

git stash 后，工作被放在了“储藏室”，这个时候假如master分支上有bug，就可以先切回master分支，在master分支上新建修改bug的分支后，合并、删除，就可以了 再回到dev分支上继续开发，用git stash list可以查看stash的内容，要恢复藏起来的内容，有两中方法：

  1. 用git stash apply 恢复，但是恢复后，stash的内容并没有删除，需要用 git stash drop 来删除stash的内容。
  2. 第二种方式是用git stash pop 来恢复，用这种方法 stash 的内容被删除了。

可以多次stash ，恢复的时候，先用git stash list 查看，然后恢复指定的stash,用命令： git stash apply stash@{0}

问题又来了，如果bug是在master分支上，那么它有bug，我们的dev分支上是不是就也有bug呢？

问题是肯定的，因为dev分支就是从master上来的，那么想要不繁琐的回到dev分支上新建改bug的分支，那么可以有一个Cherry-pick命令。可以复制一个特定的提交到当前分支： git Cheery-pick xxxxx （这个xxxxx就是在master上改bug后提交的这个操作（是一个commit操作。））

Feature分支

开发一个新功能，可以新建一个feature分支，此时如果这个分支不要了，那就使用git branch -d newbranch 删除，会发现删不掉，因为这个分支创建了没合并， 这个时候就可以强行删除： git branch -D newbranch

多人合作

查看远程库信息：用 git remote

或者用 git remote -v 显示更详细的信息。

推送分支：远程分支默认为origin ，所以推送分支就可以使用 git push origin master （意思是把master分支推送到远程库）

如果要推送其他分支，比如dev，就改成：git push origin dev

当合作者从远程clone了你的项目，它只能看到master分支，如果他要在dev分支上开发，就必须创建远程origin 的 dev 分支到本地，于是他用这个命令创建本地dev分支： git checkout -b dev origin/dev 现在就可以在dev分支上继续修改，然后时不时把dev分支推送到远程：

如果两个人提交（push）出现失败的情况： 比如他先提交，你再提交就会失败，这样就得先pull下来，然后本地merge后，再进行提交，如果本地merge有冲突，就要先解决冲突。

git pull 如果拉不下来，是因为本地dev分支没和远程建立连接，就得先建立连接： git branch –set-upstream branch-name origin/branch-name

再pull，就会成功了。

Reabase

就是把分支的提交变成一个直线，目的是使我们在查看历史提交的变化时更容易，因为分叉的提交需要三方对比。

标签管理

发布一个版本时，通常在版本库中打一个标签（tag） 创建标签: git tag 《name》 如 git tag v1.0

查看：git tag 标签和commit 挂钩 想要给之前的打标签，就可以git log 查看日志，找到那个commit 然后 git tag v0.9 xxxxx (这个xxxxx)就是commit
